# CloudMind MCP 设计准则

## 目标

CloudMind 的 MCP 不是“替 AI 做决策的黑盒应用层”，而是“给 AI 调用的知识基础设施”。

这意味着：

- 调用方 AI 负责判断当前任务应该怎样检索、怎样逐步放宽范围、怎样组合工具
- CloudMind 负责执行明确请求，并强制执行安全边界与数据边界
- 不把策略判断偷偷藏在服务端内部

一句话说，CloudMind 应优先提供 **可组合能力**，而不是过强的 **内置自动决策**。

---

## 核心原则

### 1. 决策权优先交给调用方 AI

凡是存在多种合理策略、且策略是否成立依赖当前用户意图的地方，优先交给调用方 AI 决定。

典型例子：

- 是否先用严格范围检索
- 是否允许 fallback
- 是否扩大 `topK`
- 是否优先看摘要还是正文
- 是否先搜索、再浏览、再总结
- 是否需要进行二次检索

CloudMind 不应在服务端替调用方 AI 猜测这些策略。

### 2. 安全边界必须由 CloudMind 强制执行

凡是涉及安全、权限、数据泄露风险或资源边界的地方，必须由 CloudMind 在服务端强控，不能交给调用方 AI。

当前至少包括：

- `ai_visibility`
- `summary_only`
- 未来的鉴权与访问控制
- 删除、重处理等高影响操作的授权边界
- 资源消耗上限与配额边界

调用方 AI 只能在允许范围内做策略选择，不能绕过平台边界。

### 3. 工具应暴露参数，而不是内置猜测

如果某个能力存在“严格模式”和“宽松模式”，优先通过参数显式暴露，而不是在服务端自动切换。

优先方式：

- `allowFallback`
- `allowSummaryOnly`
- `topK`
- `profile`
- 未来可能的 `preferDomains` / `avoidDomains`

避免方式：

- 服务端根据 query 自动猜当前是 coding / writing / personal 场景
- 服务端在结果不足时自动 fallback 而不告知调用方
- 服务端自动扩大范围并返回更宽结果，但调用方并不知情

### 4. 一次调用只执行一种明确策略

每个 MCP tool 调用应尽量只执行一个清晰策略，而不是在服务端内部做多轮自动尝试。

推荐模式：

1. 调用方 AI 先用更严格参数调用一次
2. 评估结果是否足够
3. 如果不足，再显式放宽参数重新调用

这样可以让调用方始终知道当前结果是如何得出的，也更不容易偏离用户原始意图。

### 5. 默认保守，放宽要显式

涉及范围扩张的参数应尽量保守默认。

例如：

- `allowFallback` 默认 `false`
- 未来如果有 `allowSummaryOnly`，也应由调用方显式打开

这样可以促使调用方 AI 先验证“严格范围是否已经足够”，再决定是否放宽。

## 设计分工

### 调用方 AI 负责

- 判断当前任务属于搜索、浏览、总结、问答还是编辑
- 判断是否需要使用 context-aware 检索
- 判断是否需要 fallback
- 判断结果是否已经足够
- 判断是否应继续扩大范围
- 判断是否会因放宽范围而偏离用户原意

### CloudMind 负责

- 执行检索、问答、读取、编辑、重处理等具体动作
- 对结果应用硬安全边界
- 保证 `deny` 内容不泄露
- 保证 `summary_only` 只暴露摘要级内容
- 返回足够清晰的结果与证据，帮助调用方继续决策

---

## 推荐工具模式

### 模式一：保留通用工具

保留这类不带策略偏置的通用工具：

- `search_assets`
- `ask_library`
- `get_asset`

它们适合作为默认能力和人类直接使用的通用入口。

### 模式二：增加 caller-driven 上下文工具

对 AI 客户端，增加带显式策略参数的工具：

- `search_assets_for_context`
- `ask_library_for_context`

这些工具可以接受：

- `profile`
- `allowFallback`
- 后续可扩展的其它显式策略参数

其中 `profile` 应视为调用方可选的 preset，而不是平台替调用方做场景判断。

### 模式三：后续拆成“检索”和“回答”两层

当需要更强控制权时，优先演进到两层模式：

1. 检索层
   - 调用方先拿候选结果
2. 回答层
   - 再基于候选结果生成 grounded answer

这样调用方 AI 才能真正控制：

- 查什么
- 查多宽
- 何时停止
- 何时重试

## `allowFallback` 的推荐语义

`allowFallback` 的语义应当保持简单明确：

- `false`
  - 仅在当前上下文偏好范围内检索
  - 不自动扩大到更宽范围
- `true`
  - 允许扩大候选范围进行检索
  - 但仍必须遵守硬安全边界

重要约束：

- CloudMind 不应在一次调用中自动先严格检索、再自动 fallback
- 是否进行第二次“更宽范围”的调用，应由调用方 AI 决定

推荐调用策略：

1. 先使用 `allowFallback=false`
2. 检查结果是否足够
3. 仅在结果不足且不会偏离用户意图时，再用 `allowFallback=true` 重试

## Tool Description 编写准则

MCP tool 描述不应只解释“参数是什么”，还应明确告知调用方 AI 推荐的使用策略。

例如对于 `allowFallback`，描述应强调：

- 默认优先关闭
- 结果不足时再重新调用
- 放宽范围时不能偏离用户意图

推荐表达方向：

> Prefer `allowFallback=false` first. If results are insufficient, you may rerun with `allowFallback=true`, but only when broader retrieval still matches the user's intent.

---

## 哪些能力适合 caller-driven 设计

适合交给调用方 AI 决定的能力：

- 搜索是否 fallback
- 检索范围是否扩大
- 是否优先正文还是摘要
- 是否先搜索再问答
- 是否需要二次检索
- 是否先浏览目录、facet、集合，再缩小范围
- 是否对资产执行更重的分析步骤

不适合交给调用方 AI 决定的能力：

- 是否突破 `ai_visibility`
- 是否读取 `summary_only` 资产的正文
- 是否绕过权限控制
- 是否越权删除、修改、重处理数据
- 是否突破平台资源与成本限制

---

## 面向后续功能的统一约束

未来新增 MCP 工具时，优先自查以下问题：

1. 这个工具是在暴露能力，还是在替调用方做隐藏决策？
2. 哪些部分属于调用方 AI 的策略空间？
3. 哪些部分属于平台必须强控的硬边界？
4. 是否可以通过显式参数替代服务端自动猜测？
5. 如果结果不足，是否应由调用方发起第二次更宽范围调用？

如果答案倾向于“策略依赖上下文”，则应优先设计成 caller-driven 工具。

---

## 当前结论

CloudMind 的 MCP 设计应遵循以下总原则：

- 平台提供能力，不替调用方思考
- 调用方 AI 决定检索策略与工具组合
- 平台强制执行安全边界与数据边界
- 默认保守，放宽必须显式
- 多轮尝试应尽量由调用方发起，而不是由服务端偷偷完成

这套准则适用于当前的 context-aware 检索，也应作为后续 MCP 功能扩展的默认设计方向。
