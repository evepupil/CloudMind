# CloudMind 前端产品设计 Spec

## 文档目的

这份文档定义 CloudMind 前端的产品化设计方向，并把抽象想法推进到
"可以直接开始做" 的程度。

文档目标有三个：

- 统一产品心智，避免前端继续按后端接口堆页面
- 给后续页面重写提供明确的信息架构和页面结构
- 让设计、组件实现、页面开发和验收有同一份参照

这份文档是前端改造的产品设计说明，不是视觉稿，也不是技术实现细节文档。

---

## 核心判断

CloudMind 不应该继续被做成一个“能调接口的后台”。

如果前端只是围绕以下内容组织：

- 上传文件
- 提交 URL
- 粘贴文本
- 资产列表
- 资产详情
- 搜索结果

那么最终产品会像一个资料管理器，而不是一个真正的个人知识库。

CloudMind 应该让用户感受到的是：

- 我在收编自己的知识，而不是录入数据
- 我在和自己的知识库协作，而不是查询数据库
- 我在沉淀长期资产，而不是操作一组接口

因此，前端设计必须优先遵循用户心智，而不是直接映射后端路由。

---

## 产品目标

CloudMind 的前端需要同时满足以下三个目标。

### 1. Calm

界面需要稳定、安静、可长时间使用。

它不应该像运维控制台那样紧张，也不应该像营销页那样喧闹。

用户应当感受到：

- 信息很多，但不乱
- 功能明确，但不吵
- 阅读和操作都很顺手

### 2. Readable

CloudMind 的核心对象是知识内容，而不是按钮或表单。

前端必须优先保证：

- 标题易识别
- 摘要易阅读
- 来源清楚
- 状态明确
- 正文层次清晰

### 3. Operable

CloudMind 是工作型产品，不是展示型产品。

前端必须让用户高频完成以下动作：

- 保存网页、文本、PDF
- 浏览和筛选资产
- 找回某条内容
- 查看处理状态和失败原因
- 基于知识库搜索
- 基于知识库发问和追问

---

## 产品气质

CloudMind 前端应该更像一个“个人知识工作台”或“研究驾驶舱”，而不是普通
SaaS 后台。

关键词：

- Calm
- Readable
- Operable
- Archive-like
- Research-oriented

希望营造的整体感受是：

- 有一点档案馆气质
- 有一点研究台气质
- 有一点编辑器产品的克制感
- 但仍然是现代 Web 应用，而不是复古界面

---

## 导航模型

CloudMind 建议采用“侧边栏主导航 + 顶部页头”的方式，而不是纯顶部导航。

原因：

- 这是一个长期使用型工作产品，而不是轻量官网
- 用户会高频在多个工作区之间反复切换
- 后续功能会自然扩展到 Ask、Capture、Activity、Settings 等多个模块
- 侧边栏更适合承载状态、快捷入口和未来扩展

建议结构：

- 桌面端：固定侧边栏 + 顶部页头 + 主内容区
- 移动端：抽屉式侧边栏 + 顶部菜单按钮 + 主内容区

侧边栏适合承载：

- 主导航
- 快捷入口
- 处理状态摘要
- 当前工作上下文

顶部页头适合承载：

- 当前页面标题
- 页内主要操作
- 次级搜索或辅助信息

## 设计原则

所有页面和组件都应遵守以下原则。

### 原则 1：页面围绕任务，而不是接口

用户不会按 `/api/ingest/*`、`/api/assets`、`/api/search` 来理解产品。

用户理解的是：

- 我想保存点东西
- 我想找到之前存过的东西
- 我想知道系统处理得怎么样
- 我想问这个知识库一个问题

因此，前端页面结构必须服务这些任务，而不是服务接口分类。

### 原则 2：资产是内容对象，不是数据库记录

资产页和详情页的设计中心应该是：

- 这条内容是什么
- 为什么重要
- 来源是什么
- 处理到哪一步了

而不是先显示一堆字段再让用户自己理解。

### 原则 3：问答必须带证据

Ask 页不能只是聊天气泡。

每条回答都必须伴随：

- 命中的资产
- 来源片段
- 可跳转的引用入口

否则 CloudMind 会退化成一个“会生成字的盒子”。

### 原则 4：采集入口必须统一

URL、文本、PDF 都属于“把信息收进来”这一个动作。

因此采集入口应统一组织成一个 Capture 体验，而不是三个孤立表单。

### 原则 5：状态反馈必须产品化

处理中的资产、失败任务、重处理入口，不应隐藏在详情页底部。

它们应该在首页、资产页、详情页等关键节点里都能被看见。

## 信息架构

前端建议按四个主区域组织。

## 1. Overview

这是首页，也是默认主入口。

用户进入产品后的第一问题不是“我能用哪些功能”，而是：

- 我的知识库现在是什么状态
- 最近新进来了什么
- 有哪些内容已经可用了
- 哪些地方需要我关注

Overview 页要解决的就是这些问题。

建议模块：

- 顶部状态总览
- 最近新增资产
- 正在处理中的资产
- 最近失败任务
- 快速保存入口
- 推荐继续追问的话题
- 活跃标签 / 来源概览

## 2. Library

这是资产浏览中心。

Library 页的职责不是“展示一张表”，而是：

- 浏览资产
- 定位资产
- 理解资产分布
- 筛选资产
- 快速进入详情和问答

建议支持多个浏览模式：

- 卡片视图
- 紧凑列表视图
- 时间流视图

## 3. Capture

这是统一采集入口。

Capture 应理解为“收件箱”而不是“上传页”。

建议收纳以下入口：

- 保存网页
- 粘贴文本
- 上传 PDF
- 未来 MCP / 浏览器插件写入状态

## 4. Ask

这是基于知识库的问答工作台。

Ask 页需要成为 CloudMind 的核心能力之一。

它不应该只是聊天界面，而应该是：

- 一个可以提出问题的区域
- 一个可以阅读回答的区域
- 一个可以查看证据的区域
- 一个可以继续追问的区域

## 页面地图

以下页面地图按“用户能理解的页面角色”来定义，而不是按代码目录定义。

## 一级页面

### `/`

页面角色：Overview

目的：

- 让用户快速理解知识库状态
- 告知最近发生了什么
- 提供下一步行动入口

### `/library` 或沿用 `/assets`

页面角色：Library

目的：

- 浏览全部资产
- 使用筛选和搜索定位资产
- 切换视图理解资产结构

备注：

- 如果短期不改路由，可以继续沿用 `/assets`
- 但 UI 和命名应逐步转向 "Library"

### `/library/:id` 或沿用 `/assets/:id`

页面角色：Asset Detail

目的：

- 阅读一条资产
- 查看来源和处理结果
- 跳转相关问答或重处理动作

### `/capture`

页面角色：Capture

目的：

- 统一完成信息采集
- 让用户在一个地方完成 URL / Text / PDF 的写入

### `/search`

页面角色：Search

目的：

- 面向关键词查找
- 未来过渡到混合检索入口

### `/ask`

页面角色：Ask

目的：

- 基于知识库问答
- 追问和回看证据

### `/jobs` 或 `/activity`

页面角色：Processing / Activity

目的：

- 集中展示处理中和失败任务
- 提供重试、查看失败原因和跳转资产入口

备注：

- MVP 阶段可以先不独立
- 但首页和详情页里必须有相关卡片或模块

## 用户流

以下用户流用于指导页面顺序、模块优先级和入口设计。

## 用户流 1：首次进入产品

目标：

- 用户第一次打开产品时，迅速理解 CloudMind 是什么
- 用户立即知道第一步该做什么

流程：

1. 进入 Overview 首页
2. 看到知识库当前状态
3. 在侧边栏和主内容区都看到明显的入口
4. 选择保存网页 / 文本 / PDF
5. 创建第一条资产
6. 返回首页，看到新资产进入处理中或已完成

设计要求：

- 首页必须有明显主 CTA
- 侧边栏里应有快捷入口
- 零数据状态必须仍有明确引导

## 用户流 2：保存一段文本

目标：

- 快速把外部知识写入系统

流程：

1. 用户进入 Capture 或侧边栏快捷入口
2. 粘贴标题和正文
3. 点击保存
4. 系统创建资产并展示状态反馈
5. 用户可立即跳到详情页或返回首页继续采集

设计要求：

- 文本采集要低阻力
- 提交成功后要有清晰反馈
- 不应该强迫用户理解处理链路细节

## 用户流 3：查找之前保存过的一条资产

目标：

- 用户知道内容存在，但不记得具体位置

流程：

1. 用户通过侧边栏进入 Library
2. 使用关键词、类型、状态筛选
3. 浏览结果列表
4. 进入资产详情页

设计要求：

- Library 搜索和筛选必须固定在高优先级区域
- 列表密度要足够高
- 卡片或列表都必须快速扫读

## 用户流 4：围绕一条资产继续提问

目标：

- 用户从资产详情进入基于知识库的问答

流程：

1. 用户阅读 Asset Detail
2. 点击 "基于这条内容继续提问"
3. 跳转到 Ask 页，并带上上下文限定
4. 查看回答与引用来源
5. 继续追问

设计要求：

- 资产详情页应有进入 Ask 的明确入口
- Ask 页要能显示“当前问题来自哪条资产”

## 用户流 5：问知识库一个开放问题

目标：

- 用户不是寻找某条资产，而是想得到带证据的回答

流程：

1. 用户通过侧边栏进入 Ask
2. 输入问题
3. 系统返回回答和引用
4. 用户查看右侧来源证据
5. 用户点击来源跳转资产详情
6. 用户回到 Ask 继续追问

设计要求：

- 回答区域和证据区域要同时可见
- 引用必须具备跳转能力
- 回答的权威感来自“可验证来源”，不是文案语气

## 用户流 6：发现某条资产处理失败

目标：

- 用户及时知道失败内容，并能恢复

流程：

1. 用户在首页、侧边栏状态区或资产页看到失败提醒
2. 进入对应资产详情
3. 查看失败原因
4. 点击重处理
5. 回到处理中状态

设计要求：

- 失败状态必须可见
- 错误说明要清楚
- 重处理操作要明显，不应藏得太深

## 视觉方向

CloudMind 不应使用默认开发者后台视觉语言。

## 色彩

建议方向：

- 主色：深墨蓝 / 深石板蓝
- 背景：暖白、轻雾灰白
- 强调色：偏青、湖蓝、灰绿
- 状态色：低饱和处理，避免高刺激红绿

视觉结果应当是：

- 有知识产品的沉静感
- 有编辑器产品的可读性
- 有研究工具的可信感

## 背景与层次

避免纯平白底。

建议：

- 使用轻微渐变背景
- 主内容区和卡片有清楚层次
- 适度使用阴影和边框，建立深浅关系

## 字体层级

建议至少建立四层文本层级：

- Hero / Page Title
- Section Title
- Body
- Meta / Status / Caption

这样用户扫读时才不会被同一字号轰炸。

## 动效

动效只服务结构理解，不服务炫技。

建议使用在：

- 页面首屏进入
- 卡片列表渐进出现
- 视图切换
- 展开引用面板
- 状态变化

## 侧边栏结构建议

侧边栏建议分成三个区域。

## 1. 品牌区

内容：

- CloudMind 标识
- 一句短定位

作用：

- 强化产品气质
- 告诉用户自己身处什么系统

## 2. 主导航区

建议项目：

- Overview
- Library
- Capture
- Ask
- Search

要求：

- 一级导航控制在 5 到 7 项
- 当前项高亮明显
- 每项文案必须是“用户理解的功能名”

## 3. 快捷与状态区

建议内容：

- Quick Save URL
- Paste Text
- Upload PDF
- Processing count
- Failed count

要求：

- 不要做成复杂二级菜单
- 只保留高价值快捷项

- Search 不应只是比 Library 少几个筛选
- 结果项要强调“为什么匹配到”
- 未来可扩成“关键词 / 语义 / 混合”切换

## 零数据与空状态策略

CloudMind 的空状态必须设计得非常认真。

因为早期用户在很多时候会看到：

- 空首页
- 空资产页
- 空搜索结果
- 空问答历史

每种空状态都要告诉用户：

- 这里是什么
- 为什么现在是空的
- 下一步最合理的动作是什么

例如：

- 首页空状态：引导保存第一条资产
- Library 空状态：提示从 Capture 开始
- Search 空状态：提示换关键词或先采集更多资产
- Ask 空状态：提示可以询问哪些问题

## 与当前代码结构的对应关系

按照当前仓库结构，后续前端设计应主要落在以下位置。

## 页面入口

- `app/routes/*`

职责：

- 路由
- 数据准备
- 页面渲染入口

## 页面与组件

- `src/features/*/components/*`

职责：

- 页面 UI
- 交互组件
- 视图组合

## 服务端逻辑

- `src/features/*/server/*`

职责：

- 页面数据获取
- action / API 组织
- 服务调用

## 重点会重写的区域

- `src/features/layout/components/*`
- `src/features/home/components/*`
- `src/features/assets/components/*`
- `src/features/search/components/*`
- 未来新增 `src/features/chat/components/*`
- 未来新增 `src/features/capture/components/*`

---

## 实施优先级建议

如果开始真正重写前端，建议按下面顺序推进。

### 第一阶段

- 确立侧边栏式 App Shell
- 改造全局视觉变量
- 重做 Overview 首页

### 第二阶段

- 重做 Library
- 重做 Asset Detail
- 建立资产相关通用组件

### 第三阶段

- 建立 Capture 页
- 统一 URL / Text / PDF 采集体验

### 第四阶段

- 正式做 Ask 页
- 把回答和证据联动做起来

### 第五阶段

- 补 Search 深化体验
- 补 Activity / Jobs 页面
- 做细节交互和状态完善

---

## 一句话结论

CloudMind 的前端不应该只是“后端接口的皮肤”。

它应该被做成一个真正的个人知识工作台：

- 能收编信息
- 能浏览资产
- 能查看状态
- 能基于知识提问
- 能长期承载用户自己的知识积累

这份文档就是后续前端重写的产品设计基线。
